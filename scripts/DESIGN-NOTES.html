1. 核心设计哲学

1.1 两阶段预观察模式

端到端路径涉及多个独立系统，直接测试全路径会遇到大量失败点，难以定位。分离为两个独立阶段：

🎯 为什么分离？

Phase 1 可无限重复运行（零副作用），Phase 2 产生真实 PR comment。分离后，基础设施问题不会浪费 Phase 2 的配置切换成本。

🔄 可复用性

任何涉及多个独立系统（扩展 + SPA + 外部 API）的端到端测试都应采用此模式。

1.2 扩展感知的检测策略

❌ 不可靠
// Content scripts 运行在隔离世界
const mcpClient = window.mcpClient;
// → undefined (主世界无法访问)
✅ 可靠
// 检测 DOM 注入信号
document.querySelectorAll('[class*="mcp-"]').length
document.querySelector('.mcp-popover')
// → DOM 可见, 不受隔离世界限制
本质性发现
Content scripts 的注入是 DOM 可见的，但其 状态是不可见的。检测扩展激活只能依赖 DOM 信号，不能依赖全局状态。

3. 错误处理模式

3.1 分层失败检测

Level 1: 连接失败

Chrome 未运行、端口未开放 → 明确错误信息

Level 2: 扩展未找到

扩展未安装或名称不匹配 → 检查扩展列表

Level 3: 页面未就绪

需要导航到 Notion AI 页面 → ensureAgentPage()

Level 4: DOM 元素缺失

SPA 未完全渲染 → 等待 reload + sleep

Level 5: 功能拦截失败

扩展未激活或 MCP proxy 未运行 → 检查 console

5. 常见陷阱与解决方案

⚠️ 导航后 DOM 检查过早

现象: 导航后立即检查 DOM，返回空结果

原因: SPA 需要时间渲染，Content script 注入需要 reload

方案: 导航 + reload 后等待 8 秒

⚠️ Submit 按钮选择器错误

现象: [aria-label="Send"] 找不到按钮

原因: Notion /chat 的 submit 按钮无 aria-label

方案: 使用 button[type="submit"]

⚠️ 隔离世界状态访问

现象: window.mcpClient 返回 undefined

原因: Content scripts 运行在隔离世界

方案: 通过 DOM 注入信号间接检测

⚠️ Agent 路由废弃

现象: 导航到 /agent/ 后找不到 AI 输入框

原因: Notion 已废弃 /agent/ 路由

方案: 使用 /chat 作为默认路由
